## Project layout

```
terraform-provider-foo/
├── main.go                          # plugin server entry point
├── internal/provider/
│   ├── provider.go
│   ├── widget_resource.go           # one file per resource
│   ├── widget_data_source.go
│   └── *_test.go                    # acceptance tests beside their resource
├── examples/                        # HCL examples, consumed by docs generation
├── docs/                            # generated by tfplugindocs, never hand-edited
└── terraform-registry-manifest.json
```

Large providers add a per-service subpackage layer so hundreds of resources stay navigable (AWS and Azure both do this).

Entry point:

```go
func main() {
    providerserver.Serve(context.Background(), provider.New("dev"), providerserver.ServeOpts{
        Address: "registry.terraform.io/yourorg/foo",
    })
}
```

## The schema packages (a common trap)

There are three different `schema` packages: `provider/schema`, `resource/schema`, and `datasource/schema`. They define similarly named types (`Schema`, `StringAttribute`, ...) but are not interchangeable. Import the one matching the context. Mixing them up is a frequent early mistake.

## Provider type

Implement `provider.Provider`: `Metadata`, `Schema`, `Configure`, `Resources`, `DataSources`. Build the API client once in `Configure` and persist it for resources and data sources to retrieve.

```go
func (p *fooProvider) Schema(_ context.Context, _ provider.SchemaRequest, resp *provider.SchemaResponse) {
    resp.Schema = schema.Schema{ // provider/schema
        Attributes: map[string]schema.Attribute{
            "endpoint": schema.StringAttribute{Optional: true},
            "token":    schema.StringAttribute{Optional: true, Sensitive: true},
        },
    }
}

func (p *fooProvider) Configure(ctx context.Context, req provider.ConfigureRequest, resp *provider.ConfigureResponse) {
    var cfg fooProviderModel
    resp.Diagnostics.Append(req.Config.Get(ctx, &cfg)...)
    if resp.Diagnostics.HasError() {
        return
    }
    // Resolve with env-var fallbacks, validate, build the client once.
    client := api.NewClient(cfg.Endpoint.ValueString(), cfg.Token.ValueString())
    resp.ResourceData = client   // retrieved in each resource's Configure
    resp.DataSourceData = client
}
```

The `Schema` method is pure declaration: it describes the `provider "foo" {}` block, reads no values, does no I/O, and must be deterministic (Core caches it and validates HCL against it before `Configure` runs). Provider config attributes are typically `Optional` rather than `Required` so users can supply values via environment variables instead; you then resolve and validate the effective value in `Configure`, raising a diagnostic if it is still missing. `Configure` is the only place to build provider-level clients, so anything later logic needs must be persisted onto `resp.ResourceData` / `resp.DataSourceData`.

## Resource (full CRUD plus import)

Implement `resource.Resource` (`Metadata`, `Schema`, `Create`, `Read`, `Update`, `Delete`), plus `resource.ResourceWithConfigure` to grab the client and `resource.ResourceWithImportState` for import.

```go
type widgetResource struct{ client *api.Client }

type widgetModel struct {
    ID      types.String `tfsdk:"id"`
    Name    types.String `tfsdk:"name"`
    Size    types.Int64  `tfsdk:"size"`
    Created types.String `tfsdk:"created_at"`
}

func (r *widgetResource) Configure(_ context.Context, req resource.ConfigureRequest, resp *resource.ConfigureResponse) {
    if req.ProviderData == nil {
        return // early validation phase; not an error
    }
    client, ok := req.ProviderData.(*api.Client)
    if !ok {
        resp.Diagnostics.AddError("unexpected provider data", "expected *api.Client")
        return
    }
    r.client = client
}

func (r *widgetResource) Create(ctx context.Context, req resource.CreateRequest, resp *resource.CreateResponse) {
    var plan widgetModel
    resp.Diagnostics.Append(req.Plan.Get(ctx, &plan)...)
    if resp.Diagnostics.HasError() {
        return
    }
    created, err := r.client.CreateWidget(ctx, api.Widget{Name: plan.Name.ValueString(), Size: plan.Size.ValueInt64()})
    if err != nil {
        resp.Diagnostics.AddError("create widget failed", err.Error())
        return
    }
    // Persist ID immediately, before anything else can fail.
    plan.ID = types.StringValue(created.ID)
    plan.Created = types.StringValue(created.CreatedAt)
    resp.Diagnostics.Append(resp.State.Set(ctx, &plan)...)
}

func (r *widgetResource) Read(ctx context.Context, req resource.ReadRequest, resp *resource.ReadResponse) {
    var state widgetModel
    resp.Diagnostics.Append(req.State.Get(ctx, &state)...)
    if resp.Diagnostics.HasError() {
        return
    }
    w, err := r.client.GetWidget(ctx, state.ID.ValueString())
    if errors.Is(err, api.ErrNotFound) {
        resp.State.RemoveResource(ctx) // gone upstream: drop it, Core plans a recreate
        return
    }
    if err != nil {
        resp.Diagnostics.AddError("read widget failed", err.Error())
        return
    }
    // Refresh EVERY tracked attribute. This is drift detection.
    state.Name = types.StringValue(w.Name)
    state.Size = types.Int64Value(w.Size)
    state.Created = types.StringValue(w.CreatedAt)
    resp.Diagnostics.Append(resp.State.Set(ctx, &state)...)
}

// Update and Delete follow the same shape: decode the right document, call the API, write or clear state.

func (r *widgetResource) ImportState(ctx context.Context, req resource.ImportStateRequest, resp *resource.ImportStateResponse) {
    resource.ImportStatePassthroughID(ctx, path.Root("id"), req, resp)
}
```

For a composite import key, parse `req.ID` (for example `strings.SplitN(req.ID, "/", 2)`) and set the parts into state. For Terraform 1.12+ prefer resource identity (see `advanced-primitives.md`), which gives structured, typed import instead of stringly-typed key parsing.

## Diagnostics: the `Append(...)` / `HasError()` pattern

The framework does not use Go `error` returns for user-facing problems. `Get`/`Set` return `diag.Diagnostics` (a slice; each entry has severity, summary, detail, and an optional attribute path). The idiom:

```go
resp.Diagnostics.Append(req.Plan.Get(ctx, &plan)...) // ... spreads the returned slice into the variadic Append
if resp.Diagnostics.HasError() {                      // Append does not stop execution
    return
}
```

Two reasons this exact shape matters. `Append` only accumulates; without the guard you proceed with a partially-decoded struct and make API calls with garbage. And you check `HasError()` rather than `len(resp.Diagnostics) > 0` because warnings are legitimate and must not halt the operation. This is the framework's substitute for `if err != nil { return err }`, and it follows nearly every `Get`, `Set`, and nested-object call.

## Schema, types, and the model struct

A schema is a map of attribute names (the literal HCL argument names) to attribute definitions, plus an optional `Blocks` map for nested blocks. Attribute values use framework wrapper types (`types.String`, `Int64`, `Float64`, `Bool`, `Number`, `List`, `Set`, `Map`, `Object`, `Dynamic`) mapped to a Go struct via `tfsdk` tags.

The `tfsdk` tag on each struct field must exactly match the schema attribute name; `req.Plan.Get(ctx, &m)` decodes against the schema into the struct, and a mismatch is a runtime error, not a compile error. Keep them in sync by hand.

Attribute behaviors:

- **Required**: user must set it.
- **Optional**: user may set it.
- **Computed**: provider sets it (an output or a default).
- **Optional + Computed**: user may set it, otherwise the provider/API picks a value. This is where most unknown-handling subtlety lives.

Always add `Description`/`MarkdownDescription` (these feed generated docs). Mark secrets `Sensitive: true` (redaction only; see `secrets-and-sensitive-data.md`).

## Plan modifiers and validators

**Plan modifiers** shape the proposed plan at plan time. The essential three:

- `RequiresReplace()` — changing this attribute forces destroy-and-recreate. Use it for anything the API cannot change in place.
- `UseStateForUnknown()` — for a computed attribute stable across updates, carry the prior state value forward instead of showing "known after apply" every plan. Prevents a large class of noisy diffs.
- `RequiresReplaceIf(...)` and custom modifiers for conditional behavior.

They exist at the attribute level and resource level (`ModifyPlan`).

**Validators** run before plan and produce clean diagnostics. Use the `terraform-plugin-framework-validators` module (`stringvalidator.LengthAtLeast`, `int64validator.Between`, conflicting/required-together validators) rather than hand-rolling checks inside CRUD methods.

## Data sources

Implement `datasource.DataSource` (`Metadata`, `Schema`, `Read`), optionally `Configure`. Data sources have no lifecycle and own no state; both "get" and "list" are pure reads that run every plan. The difference between a singular get and a plural list is only the schema and what `Read` does:

- **Get (singular)**: require a unique key (id/name), fetch one object, populate a flat model.
- **List (plural)**: accept filter inputs, fetch a collection, populate a `ListAttribute`/`SetNestedAttribute`.

Ship them as two separate data sources with distinct type names rather than one overloaded source, because the output shapes (object vs collection) must stay stable for Terraform's type system. See `naming-conventions.md` for how to name the pair. Do not confuse a plural data source with the List Resources primitive (`advanced-primitives.md`), which is a different feature for discovering unmanaged infrastructure.
